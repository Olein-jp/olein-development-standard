# 00 Philosophy

このドキュメントは、Olein Development Standard（ODS）の思想を定義します。

ODS に従う開発では、実装方法よりも先に、何を大切にして判断するかを明確にします。

## 1. User First

プロダクトは、技術のためではなく、ユーザーの課題を解決するために作ります。

MUST:
- ユーザーにとっての価値を最優先に考える。
- 技術的に面白いだけの機能を優先しない。
- 実装都合だけで UX を悪化させない。

SHOULD:
- 誰の、どの課題を解決するのかを言語化する。
- 機能追加時は、ユーザー価値を説明できる状態にする。

## 2. MVP First

最初から完成版を目指さず、小さく価値のある形で作ります。

MUST:
- MVP のスコープを明確にする。
- まず動く形にする。
- 現在のマイルストーンを途中で広げない。

SHOULD:
- Core だけでも価値がある状態を目指す。
- 公開後に改善できる余地を残す。

MUST NOT:
- 完成前に過剰な再設計を繰り返す。
- 将来使うかもしれないという理由だけで機能を追加する。

## 3. UX First

ユーザーに使われることを前提に設計します。

MUST:
- ユーザーが迷わず使えることを重視する。
- 開発者都合だけで画面や操作を複雑にしない。

SHOULD:
- 初めて使う人でも理解しやすい表現を選ぶ。
- エラー、空状態、確認メッセージを丁寧に設計する。
- アクセシビリティをできるだけ考慮する。

## 4. Maintainability

短期的な速さだけでなく、長く保守できることを重視します。

MUST:
- 未来の自分や他の開発者が理解できる構成にする。
- 不必要に複雑な設計を避ける。
- 重要な設計意図はドキュメントに残す。

SHOULD:
- 現実的で読みやすい実装を選ぶ。
- 抽象化よりも、変更しやすさを重視する。

## 5. Simplicity

シンプルさは重要な品質です。

MUST:
- 不要な機能、不必要な設定、不必要な抽象化を避ける。
- 説明できない複雑さを入れない。

SHOULD:
- 小さな単位で作る。
- 読みやすく、追いやすい構成にする。

## 6. YAGNI

将来必要になるかもしれない、という理由だけで実装しません。

MUST:
- 現在必要なものを優先する。
- 将来案と現在の実装を分ける。

MAY:
- 将来の拡張を妨げない設計にする。

MUST NOT:
- 未確定の将来機能のために、現在の MVP を複雑にする。

## 7. Documentation as Source of Truth

チャットや口頭のやり取りではなく、ドキュメントを正本とします。

MUST:
- 決定事項をドキュメントに反映する。
- 仕様変更時は変更理由を残す。
- ドキュメントと実装が矛盾した場合は、矛盾を明示して解消する。

SHOULD:
- 実装後にドキュメントを更新する。
- ドキュメントは完璧さよりも、現在の判断を追えることを重視する。

## 8. AI Collaboration

AI は開発に参加する協働者として扱います。

MUST:
- AI は ODS の判断基準に従う。
- 小さな判断は自律的に進めてよい。
- 大きな判断は人間に確認する。

SHOULD:
- AI は理由を持って提案する。
- AI は現在のスコープを守る。
- AI は新しいアイデアを思いついても、現在の完成を妨げない。

## 9. Efficiency First

AI や開発作業では、品質を維持しながら無駄を減らします。

MUST:
- 必要な情報だけを読む。
- 必要十分な出力にする。
- 同じ説明や同じ調査を繰り返さない。

MUST NOT:
- トークン削減や作業短縮を理由に、必要な品質確認を省略する。

## 10. Complete First, Improve Later

より良い案が見つかっても、現在のスコープを壊してはいけません。

MUST:
- まず現在の成果物を完成させる。
- 改善案は次のマイルストーンへ回す。
- 完成前に構成や方針を広げ続けない。

この原則は、ODS 自体の策定にも適用します。
