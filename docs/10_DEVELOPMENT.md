# 10 Development

このドキュメントは、Olein Development Standard（ODS）における汎用的な開発ルールを定義します。

ODS v0.1 では、特定の言語、フレームワーク、CMS に依存しない Web プロダクト開発の基本ルールのみを扱います。

## 1. Scope Control

開発では、現在のスコープを守ることを重視します。

MUST:
- 作業開始前に、今回完成させる範囲を明確にする。
- 作業中に新しいアイデアが出ても、現在のスコープを勝手に広げない。
- 改善案は現在の作業を止めず、次のマイルストーンへ回す。

SHOULD:
- 小さく完成させてから改善する。
- 完成条件を簡潔に定義する。

MUST NOT:
- 完成前に過剰な再設計を始める。
- 将来案を現在の MVP に混ぜる。

## 2. MVP Development

MVP は、単に機能が少ない状態ではありません。

MVP は、Core だけでもユーザーに価値を提供できる最小の形です。

MUST:
- MVP の対象ユーザーと解決する課題を明確にする。
- Core だけで価値が伝わる状態を目指す。
- 不要な機能を追加しない。

SHOULD:
- 最初のリリース後に改善できる余地を残す。
- ユーザーが最初に体験する流れを優先して作る。

## 3. User Experience

UX は後回しにする装飾ではなく、プロダクト品質の一部です。

MUST:
- ユーザーが迷わず使える流れを設計する。
- 操作、文言、エラー、空状態を軽視しない。
- 開発都合だけでユーザー体験を悪化させない。

SHOULD:
- 初めて使う人にも分かりやすい言葉を使う。
- 重要な操作には確認やフィードバックを用意する。
- アクセシビリティをできるだけ考慮する。

## 4. Maintainable Implementation

実装では、技術的に美しいことよりも、現実的に保守しやすいことを優先します。

MUST:
- 読みやすく、追いやすい構成にする。
- 不必要な抽象化を避ける。
- 重要な設計意図はドキュメントに残す。

SHOULD:
- 小さな単位で実装する。
- 変更しやすい構成を選ぶ。
- 既存の一般的な慣習に従う。

MUST NOT:
- 技術的な美しさだけを理由に複雑な構成を採用する。
- 将来必要になるかもしれないという理由だけで仕組みを増やす。

## 5. External Dependencies

外部ライブラリやサービスは、価値がある場合は利用してよいです。

MUST:
- 依存を追加する理由を説明できる状態にする。
- 依存によって保守性やセキュリティが大きく悪化しないか確認する。

SHOULD:
- 広く使われ、保守されている依存を選ぶ。
- 不要になった依存は削除する。

MUST NOT:
- 小さな処理のために過剰な依存を追加する。

## 6. Documentation

チャットではなく、ドキュメントを正本とします。

MUST:
- 決定事項をドキュメントに反映する。
- 仕様変更時は理由を残す。
- 実装とドキュメントが矛盾する場合は、矛盾を明示する。

SHOULD:
- 実装後にドキュメントを更新する。
- 完璧なドキュメントよりも、現在の判断を追える状態を優先する。
- 仕様、未決事項、将来案を分けて書く。

## 7. Quality Checks

品質確認は、プロダクトの信頼性を保つために行います。

MUST:
- 公開前に最低限の動作確認を行う。
- 重要な機能は手動または自動で確認できる状態にする。
- 重大な不具合を放置して公開しない。

SHOULD:
- Lint、型チェック、静的解析、自動テストを導入する。
- プロダクトの規模に応じて品質チェックを強化する。
- 公開前チェック用の Issue またはチェックリストを用意する。

## 8. Security

セキュリティは公開前に必ず確認します。

MUST:
- 認証、権限、入力検証、出力エスケープを確認する。
- 個人情報や機密情報を不適切に扱わない。
- 公開前にセキュリティ確認のタスクを作成する。

SHOULD:
- MVP 段階でも最低限の安全性を確保する。
- 公開前により厳格な確認を行う。

## 9. Git and Change Management

変更は追跡可能であるべきです。

MUST:
- 変更理由が分かる単位でコミットする。
- 既存仕様を変更する場合は理由を残す。
- 公開後の互換性に影響する変更は慎重に扱う。

SHOULD:
- 初期開発では柔軟に進める。
- プロダクトが育ってきたら Issue や Pull Request を活用する。
- 正式リリース後は Breaking Change を原則避ける。

## 10. AI Collaboration

AI は、ODS に従って開発を支援します。

MUST:
- AI は現在のスコープを守る。
- AI は重要な設計変更を勝手に行わない。
- AI は必要以上に長い出力や不要なファイル読み込みを避ける。

SHOULD:
- 小さな判断は ODS に基づいて仮決めして進める。
- 大きな判断は人間に確認する。
- 判断理由を簡潔に示す。

MUST NOT:
- 新しいアイデアで現在の完成を妨げる。
- トークン削減を理由に必要な確認を省略する。

## 11. Versioning and Release

プロダクトは小さく育てます。

MUST:
- MVP 段階では変更しやすい状態を保つ。
- 正式リリース後は互換性を重視する。

SHOULD:
- `0.x` で育て、安定後に `1.0.0` とする。
- リリースごとに変更点を記録する。

## 12. Complete First, Improve Later

開発中により良い案が出ても、まず現在の成果物を完成させます。

MUST:
- 現在の成果物を完成させることを優先する。
- 改善案は現在の作業を止めず、次の改善候補として扱う。
- スコープを広げる場合は確認を取る。

このルールは、ODS 自体の策定にも適用します。
